# レビューコメント回答ガイドライン

## 基本方針

レビューコメントに対応する際は、以下の手順に従って作業を進めてください。

### 1. レビューコメント分析・方針整理（必須事前ステップ）

レビューコメントを受けた際は、**作業開始前**に以下を必ず整理してください：

#### 1.1 対応ファイル・内容の明確化
- **具体的なファイル名**：どのファイルを修正するか
- **更新内容**：そのファイルの何をどのように変更するか
- **修正範囲**：変更が影響する他のファイルはあるか

#### 1.2 対応方針の文書化
レビューコメント対応時は、以下の形式で方針を明文化してから作業に取り組む：
```markdown
## レビューコメント対応方針

### 対応対象
- ファイル名：[具体的なファイルパス]
- 修正内容：[具体的な修正内容]
- 影響範囲：[関連するファイルや機能]

### 実装方針
- [どのような方法で修正するか]
- [考慮すべき設計原則や制約]
- [テストの追加・修正が必要か]
```

### 2. タスク定義の具体化

抽象的なタスク表現（例：「ドキュメント更新」）は避け、具体的に定義してください：

#### 悪い例
- ドキュメント更新
- テスト修正
- リファクタリング

#### 良い例
- README.mdの設定手順セクションを最新のコマンドに更新
- UserService.validate()メソッドの異常系テストをAbendディレクトリに移動
- PaymentCalculatorクラスの責務分離：手数料計算ロジックを新クラスに抽出

### 3. 包括的な対応の実施

#### 3.1 段階的改善の回避
- 複数回の小さな修正よりも、一度の包括的な対応を優先
- 関連する修正点は可能な限り同時に対応
- 設計変更の場合は、最終形を見据えた実装を実施

#### 3.2 根本的解決の重視
- 指摘された問題の表面的な修正ではなく、背景にある設計原則を理解
- 同種の問題が他に発生していないか確認
- 設計原則に基づいた根本的解決を実施

### 4. システム理解の見直し

#### 4.1 主要機能の理解確認
- レビューコメントを機に、システムの主要機能についての理解を再確認
- 機能の重要度と影響範囲を正しく評価
- 不明な点があれば事前に調査・確認

#### 4.2 技術的修正と理解修正の両方実施
- コードの技術的修正だけでなく
- システム理解や設計思想の見直しも含めた包括的な対応
